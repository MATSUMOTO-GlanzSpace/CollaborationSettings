# コラボレーションガイドライン

## このリポジトリについて

このリポジトリは、共同開発（コラボレート開発）のための資料やベストプラクティスを提供することを目的としています。

## 編集ポリシー

### 誰が編集できるか

- ✅ **リポジトリオーナー（作成者）**: すべての編集が可能
- ❌ **一般ユーザー**: 直接の編集は不可

### なぜこの制限があるのか

1. **品質管理**: 内容の一貫性と品質を保つため
2. **正確性の維持**: 誤った情報の混入を防ぐため
3. **構成の統一**: ドキュメント構成を統一するため

## 貢献する方法

編集権限がない場合でも、以下の方法で貢献できます：

### 1. Issueの作成

誤りを発見した場合や改善提案、質問、作業依頼、調査事項がある場合は Issue を作成してください。

1. Issuesタブを開く
2. 「New Issue」をクリック
3. 適切なテンプレートを選択
4. テンプレートに沿って具体的に記述
5. 提出

### 2. Fork & Pull Request

コードや文書の変更を提案する場合：

1. リポジトリをフォーク（右上の「Fork」ボタン）
2. フォークしたリポジトリで変更を加える
3. 変更をコミット
4. オリジナルリポジトリにPull Requestを作成
5. オーナーによるレビューを待つ

### 3. ディスカッション

- GitHubのDiscussionsが有効な場合、そこでアイデアを共有
- または、Issueでディスカッションを開始

## 推奨される貢献内容

以下のような貢献を歓迎します：

### バグ報告
- リンク切れの報告
- 誤った情報の指摘
- 古くなった情報の指摘
- 手順や例が動かない／再現しない など

### 改善
- 説明の明確化（誤解を減らす）
- 新しい例の追加・改善
- 図や表の追加
- 構成の見直し（読みやすさ向上）
- 品質向上（リファクタ、運用改善、性能/UX改善など）

### 質問
- 進め方、判断基準、運用ルールの確認
- 「Issue と PR のどちらが適切か」などの相談

### タスク
- やることが明確な作業（更新作業、整理、設定変更、移行など）

### 調査
- 原因調査、技術調査、検証（Spike/PoC）
- 方針決定のための情報収集

## Issue作成ルール（分類とタイトル）

Issue は次の 5 種類に分類します。**タイトルは接頭辞で分類を明確化**してください。

- **バグ**：`バグ: <要約>`
- **改善**：`改善: <要約>`
- **質問**：`質問: <要約>`
- **タスク**：`タスク: <要約>`
- **調査**：`調査: <要約>`

> 可能であれば、タイトル末尾に対象を付けると探しやすくなります。  
> 例：`バグ: リンクが404になる [COLLABORATION.md]`

## Issueテンプレートについて

Issue はテンプレートを用意しています。作成時に該当テンプレートを選び、項目を埋めてください。

- 配置場所：`.github/ISSUE_TEMPLATE/`
- 種類：バグ / 改善 / 質問 / タスク / 調査

## レビュープロセス

Pull Requestが作成された場合：

1. **オーナーによるレビュー**
   - 内容の正確性確認
   - 既存ドキュメントとの整合性確認
   - コーディングスタイル確認（該当する場合）

2. **フィードバック**
   - 必要に応じて変更依頼
   - コメントでのディスカッション

3. **マージ**
   - 承認されたらマージ
   - 貢献者に感謝のメッセージ
